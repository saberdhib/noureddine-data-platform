## Section B — What's been fixed in this audit ✅

### Fix A — Airflow image inherits dbt + ML deps (CRITICAL, inherited §D.1) — `fix(airflow): ...`
- New `infra/airflow/Dockerfile` (`noureddine-airflow:local`): **dbt-core/dbt-postgres 1.8.2 in an
  isolated venv** `/opt/dbt-venv` (so dbt's pins never clash with Airflow's), plus ML/monitoring
  libs (`requirements-airflow.txt`: lightgbm, pandas, numpy, scikit-learn, shap, scipy, joblib,
  matplotlib, pyarrow, evidently — ranges so pip resolves against Airflow 2.9 constraints).
- `infra/docker-compose.yml`: the `airflow` service now **builds** that image (was `image:`);
  dropped the fragile runtime `_PIP_ADDITIONAL_REQUIREMENTS`; the `dbt` mount is now **read-write**
  so `dbt deps`/`target` can be written; added `DBT_BIN`/`DBT_PROJECT_DIR`/`DBT_PROFILES_DIR`.
- `dags/ingest_orders.py` (the permitted §D.1 touch): runs `"$DBT_BIN deps && $DBT_BIN build"`
  via the isolated venv binary.
- **Verified statically** (no Docker here): `docker compose config` passes; `ingest_orders` parses
  with the new command. The image build + a live DAG run must be verified on a Docker host (§C/§D).

### Fix B — DPIA #2 enforcement test — `feat(ml): ...`
- New `ml/tests/test_no_pii_in_features.py`: asserts the trained `feature_names` + matrix are
  disjoint from every forbidden individual-level column, that every feature is calendar/aggregate
  only, and that the `assert_no_pii` guard raises on injected PII.
- **Now passes:** `pytest ml/tests/` → **12 passed** (was 9). Runs in the `ml-test` CI job, so the
  DPIA #2 commitment is enforced on every push.

### Fix C — CI `dbt deps` Elementary version — `fix(ci): ...`
- The latest `main` run (27416611754) was **failure**: after the Bloc 2/3 audit fixed
  `dbt-postgres` to 1.8.2, the **next** step `dbt deps` failed because `packages.yml` pinned
  Elementary `0.14.3` (does not exist; the hub line jumps 0.14.1 → 0.15.0).
- `packages.yml` now uses a range `[">=0.14.0","<0.15.0"]` → `dbt deps` resolves an existing 0.14.x.

## Section C — Manual testing guide

> Local prereq (what this audit used): PostgreSQL on `localhost:5432`, the Bloc 3 simulator + dbt
> already run so `gold` is populated. Docker prereq: `cp .env.example .env && bash infra/scripts/up.sh`.
> Export for local runs:
> `export POSTGRES_HOST=localhost POSTGRES_USER=noureddine_user POSTGRES_PASSWORD=change_me_postgres POSTGRES_DB=noureddine API_KEY=noureddine-dev-key`

1. **Train the model**
   ```bash
   python ml/src/train.py
   ls -l ml/models/current.pkl ml/models/shap_summary.png ml/models/metrics.json
   readlink ml/models/current.pkl     # -> model_<timestamp>.pkl (atomic-promotion symlink)
   ```
   Expect: model saved + promoted, SHAP + metrics written. Failure `No rows`: run the simulator + `dbt build` first.

2. **Verify no PII (DPIA #2)**
   ```bash
   pytest ml/tests/test_no_pii_in_features.py -v
   ```
   Expect: 3 passed. Failure → a PII column reached the feature matrix; inspect `features.FEATURE_COLUMNS`.

3. **Start the API**
   ```bash
   docker compose -f infra/docker-compose.yml up -d api          # or: uvicorn main:app --app-dir api --port 8000
   docker inspect --format '{{.State.Health.Status}}' noureddine_api    # healthy
   ```

4. **`/health` + `/model-info`**

   ```bash
   curl -s localhost:8000/health ; echo ; curl -s localhost:8000/model-info
   ```
   Expect: `{"status":"ok",...}` and JSON with `version`, `global_mape`, `categories`, `feature_names`.

5. **`/predict` without key → 401**

## Section D — What still needs work IN the repo ⚠️

1. **Airflow image build must be verified on a Docker host (Fix A).** The Dockerfile + compose are
   static-validated only (no Docker here). On build, confirm `evidently>=0.7` resolves against
   Airflow 2.9's `numpy<2` constraint; if it does not, install the monitoring deps into a **second
   isolated venv** and run `monitor_model` via `ExternalPythonOperator` (the dbt venv already proves
   the pattern). Verify: `docker compose exec airflow /opt/dbt-venv/bin/dbt --version` and
   `python -c "import lightgbm, evidently"`.
2. **Model quality is volume-sensitive (FINDING, not a bug).** On the **reduced-volume** audit data
   (`SIM_BASE_DAILY_ORDERS=9`, 7 categories) per-category-day counts are small, so global **MAPE ≈
   0.77** (sMAPE 0.54) and the **nominal** `monitor_model` breaches the `MAPE_THRESHOLD=0.30`. This
   is MAPE's instability at small denominators, not a code defect (the model trains, predicts and
   promotes correctly). Recommendations: (a) run the simulator at **full volume**
   (`SIM_BASE_DAILY_ORDERS≥15`) for representative metrics and a clean nominal-vs-breach demo; (b)
   consider making **sMAPE** the headline/threshold metric; (c) optionally raise `MAPE_THRESHOLD` for
   sparse categories. The forced-drift path still separates clearly (drift 0.0 → 0.8).
3. **CI `dbt deps` / Elementary verified only via CI.** The hub is blocked in this env; the range
   pin (Fix C) is confirmed by the `dbt-validate` job on the pushed branch, not locally.
4. **Bloc 4 DAGs not executed in a live scheduler here.** Task callables + DAG structure were
   verified; a real Airflow run depends on Fix A (Docker host).

## Section F — Recommended jury Q&R prep

| Question | Answer (reference) |
|----------|--------------------|
| Why LightGBM not Prophet? | Explainable (SHAP), seconds to train, strong on event-driven peaks — **ADR-0009** + `shap_summary.png`. |
| How do you guarantee no personal data in the model? | DPIA #2 — features are calendar + per-category aggregates only; **enforced** by `ml/tests/test_no_pii_in_features.py` (fails the build on leak). |
| How is the model retrained? | `retrain_model` (Sundays 02:00) + `monitor_model` (daily, drift-triggered via `TriggerDagRunOperator`); promotion is **atomic** (timestamped file → validate → swap `current.pkl` symlink). |
| What if the new model is worse? | MAPE gate (promote iff `new ≤ current × 1.05`), else hold; prior versions kept in `ml/models/` for rollback — logged in `monitoring.retrain_events`. |
| How do you detect drift? | Evidently (data + target drift) → `monitoring.model_metrics`; env thresholds `DRIFT_THRESHOLD`/`MAPE_THRESHOLD`; surfaced in Grafana "Model Health". |
| Why FastAPI not Flask? | Async, native OpenAPI, Pydantic type-safety — **ADR-0010**; `/docs` live. |
| Why Streamlit not Power BI? | Zero-licence + tight Python/model integration; Streamlit calls the API (never loads the model) — **ADR-0012**; Power BI/Metabase noted as future option. |
| Why no Kubernetes? | Disproportionate for a 30-FTE PME on a single host — **ADR-0013** (consistent with Bloc 2 ADR-0002). |
